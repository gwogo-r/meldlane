Why Meldlane

Not Jira/Linear with an AI feature added on — a tracker built around the fact that agents do real work now.

🤝

Agents are equal assignees

One Member type for Roman and for Claude Dev. One Task type, assigned to either. Not a separate "AI mode" bolted on the side.

📊

Capacity that reflects real cost

Humans show story points. Agents show tokens, split into real API billing vs subscription cost-equivalent — so you never confuse the two.

⚙️

Real execution, not planning theater

Agent-assigned tasks run as actual Claude Code / Codex subprocesses in an isolated workspace copy, and produce real file changes — verified live.

🎙️

Tasks come from your meetings

Mic + system audio (or a file) in, a transcript with speaker labels out, an LLM extracts action items with assignee and story-point guesses.

🔄

Synced to Plane, idempotently

The tracker core isn't reinvented. Tasks push to a real Plane instance and re-sync without creating duplicate issues.

⚖️

Honest about the gaps

Plane doesn't support agents as native assignees, so agent name/cost/story points go into the issue description as text. See "Status" below for what's not solved yet.

Built as a modular toolchain

Audio capture/transcription, LLM calls, and Plane sync are each their own published, independently useful package. This repo is the orchestrator that wires them together.

packagewhat it does
meldlane-transcribe Mic + system audio (WASAPI loopback on Windows, no VB-Cable) or a file in, a structured transcript with speaker labels out.
llm-gateway One call for OpenRouter/OpenAI chat completions and for running Claude Code / Codex as real subprocess agents, with a single TokenUsage model tracking which kind of billing applies.
meldlane-tasks Idempotent Task → Plane issue sync — push the same task twice, get one issue, not two.

Status

A working MVP, not a polished product yet.

⚠ What's verified vs. what isn't

Verified live: meeting → task extraction, real agent execution (actual files written), idempotent Plane sync, and capacity accounting all run against real data, not mocks.

Not yet: the web UI hasn't been checked in an actual browser (API + build verified only); the Telegram confirm-gate is written but untested live — real agent execution currently skips confirmation, in disposable working copies only.

Known limitation: a CLI agent's working directory is a convention, not a filesystem jail — there's no real sandbox yet. Handled procedurally for now, not solved technically.

Install

Python 3.12+. Clone, install, point it at your LLM/Plane keys.

# backend
pip install -r requirements.txt
cp .env.example .env
# see your team, extract tasks from a sample meeting, check capacity
python main.py members
python main.py extract samples/standup.txt
python main.py capacity
# web UI
python -m uvicorn api.app:app --reload      # port 8000
cd frontend && npm install && npm run dev   # port 5173
