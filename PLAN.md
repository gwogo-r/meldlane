# Meldlane — План запуска

Цель первого спринта: **замкнуть конвейер «митинг → задачи» на минимуме**. Записать один реальный синк (или взять готовый транскрипт) и получить из него список задач — с оценкой и назначением на человека или агента. Не весь продукт, а работающий срез, который уже показывает суть Meldlane.

Каждый шаг — с проверяемым итогом. Не идём дальше, пока предыдущий не даёт результат руками.

**Итог:** `python main.py extract samples/standup.txt` печатает задачи с оценкой и назначением; в `token_usage` залогированы токены и стоимость прогона. Это ядро ценности — смотрим глазами, адекватные ли задачи. *(Требует ключ LLM в `.env`.)*

## Шаг 3 — AudioCapture + Whisper (день)
- [x] `capture/audio.py` (mic + system audio через VB-Cable) + `capture/transcriber.py` (Whisper).

**Итог:** `python main.py capture` пишет живой звук в WAV, `python main.py transcribe <wav>` даёт транскрипт → тот же extractor. Полный цикл **аудио → задачи** без ручного ввода. Проверено: запись 3 сек с mic + распознавание через Whisper (`base`) отработали. System audio (VB-Cable) в коде поддержан, но кабель физически не установлен — дождётся установки на машине.

## Шаг 4 — Plane + агент как assignee (день) — ОТЛОЖЕН
- [ ] Поднять Plane self-hosted (docker-compose), `tracker/plane.py` (клиент API), расширить модель ассайни (агент = участник).

**Статус:** сознательно отложено 2026-07-01. Официальный self-host CE — 13-сервисный стек (Postgres/Redis/RabbitMQ/MinIO/web/api/admin/space/live/nginx), сборка из исходников 10-20+ мин, несколько ГБ диска, конфликт портов 80/443 на локальной машине. Решение: пробуем поднять на внешнем VPS (не локально), когда VPS будет готов — так снимаются и порты, и нагрузка на рабочую машину. До этого момента задачи живут в локальном SQLite (`storage/db.py`), интерфейс `tracker/plane.py` подключим отдельно, не блокируя остальной план.

**Итог (когда сделаем):** задачи из митинга появляются как issue в Plane; ассайни — человек **или** агент.

## Шаг 6 — ConfirmGate: Telegram-подтверждение (полдня)
- [x] `agents/confirm.py` (aiogram).

**Итог:** перед необратимым действием агент шлёт «подтвердить?» в Telegram и ждёт человека. Код написан и импортируется; **живьём не проверено** — нужны `TELEGRAM_BOT_TOKEN`/`TELEGRAM_CHAT_ID` в `.env`. Ручная проверка: `python -m agents.confirm`.

## Шаг 7 — AgentExecutor (день+)
- [x] `agents/executor.py` — реальное исполнение через CLI-агентов.

**Итог:** `agent.provider` в `team.yaml` определяет режим: `openrouter` — planning-only (старое поведение), `claude-code`/`codex` — **реальное исполнение** через `agents/cli_runner.py` (подпроцесс `claude -p .../codex exec ...`), списывается с подписки, а не по цене за токен. Проверено вживую:
- Claude Code CLI: явная задача «создай файл hello.txt» в изолированной scratch-папке — файл реально создан на диске, usage/cost взяты из `total_cost_usd` в JSON-выводе.
- Codex CLI: аналогично, `codex exec --json` — NDJSON-поток событий распарсен, токены есть, `cost_usd=0` (подписка ChatGPT не даёт $-цифру).
- `run-task 8b080326ece2 --no-confirm` (задача «Сверстать лендинг») прошёл end-to-end через `claude-code`, статус → `testing`, стоимость видна в `capacity` ($0.033 суммарно у Claude Dev).
- Модель/эффорт выбираются по `story_points` задачи (`select_effort`): ≤2 SP → haiku/low, ≤5 → sonnet/medium, >5 → opus/high.
- Исполнение идёт в изолированной `out/agent_workspace/` (не в самом репозитории Meldlane) — намеренно, чтобы агент не трогал наш код по умолчанию.

**Технические грабли по пути (Windows):** `asyncio.create_subprocess_exec` не умеет напрямую запускать `.cmd`-скрипты (npm кладёт `codex.cmd`) — WinError 193; чинится запуском через `cmd.exe /c`. Также в npm-папке лежит одноимённый файл без расширения (шелл-скрипт для WSL/git-bash) — на Windows его нельзя находить раньше `.cmd`.

**Честная граница:** агент не обязательно поймёт задачу как "нужно писать файлы", если промпт сформулирован как описание, а не явная команда — качество экстрактора задач (Шаг 2) должно генерировать более императивные формулировки для CLI-агентов. ConfirmGate перед реальным исполнением не проверен живьём (нет Telegram-токена) — сейчас тестируем только через `--no-confirm`.

**Дообкатка на реальной задаче для Codex Dev (тот же день):** назначили Codex Dev задачу «написать unit-тесты для compute_capacity» через полноценный `extract` → `run-task`. Нашли и починили два реальных бага:
1. `agents/executor.py` работал в **пустой** изолированной `out/agent_workspace/` — агенту физически нечего было тестировать. Добавлена `sync_workspace()`: свежая копия репозитория (без `.git`/`out`/`.env`/БД) перед каждым запуском — агент видит реальный код, но правит только копию.
2. Codex через `cmd.exe /c` на многострочном промпте (заголовок + описание с `\n\n`) терял флаг `--json` — cmd.exe рвёт командную строку на переносах строк даже внутри кавычек, и Codex тихо откатывался на human-readable вывод. Токены парсились как 0 без явной ошибки. Фикс: схлопывать `\n` в аргументах перед сборкой команды для `cmd.exe`.
3. Codex `exec` по умолчанию в `sandbox=read-only` — не может писать файлы, даже сгенерировав правильный код (проверено: ответил кодом теста текстом, ничего не создал на диске). Добавлен `--sandbox workspace-write` — безопасно, т.к. всегда работает в изолированной копии.

После фиксов — **полный успех**: Codex Dev реально создал `tests/test_capacity.py` с 4 осмысленными тестами на `compute_capacity`, все 4 прошли (`pytest tests/ -v` — 4 passed). Файл принят в репозиторий как есть.

# Фаза 2 — разбиение на независимые сервисы (решение 2026-07-02)

Спринт 1 (фаза 1 выше) доказал конвейер целиком в одном репозитории. Дальше проект режется на **независимые сервисы**, каждый выгружается на GitHub отдельным репозиторием и полезен сам по себе. Работаем итеративно: одна итерация = один сервис доведён до рабочего публичного релиза. Полной системой не упарываемся — важные функциональные блоки по очереди.

## Архитектурные решения (приняты в обсуждении)

- **Слой 4 (LLM-структурирование) — библиотека, не сервис.** `llm-gateway`: провайдеры (OpenRouter/OpenAI/claude CLI/codex CLI), строгий JSON-выход, единый `TokenUsage`. Импортируется остальными сервисами — так сохраняется сквозной учёт токенов/$ (фирменная фича capacity). Ядро уже написано: `agents/cli_runner.py` + `metrics/logger.py`.
- **Диаризация спикеров — каналами, не ML.** Mic и system audio пишутся отдельными потоками: mic-дорожка = «я», system = «собеседники» — бесплатно и без настройки. Полная диаризация (pyannote, torch + HF-токен) — только опциональный extra для загруженных файлов с одной дорожкой.
- **Захват звука должен работать у всех, не только на этой машине.** Stereo Mix — удача конкретного железа (Realtek, выключен по умолчанию, не ловит Bluetooth/USB-наушники; на macOS loopback нет вообще). Универсальный каскад: WASAPI loopback (Windows, из коробки) → BlackHole (macOS, `brew install blackhole-2ch`, автодетект + подсказка) → Stereo Mix/VB-Cable (fallback) → только mic + предупреждение. Плюс команда `doctor` и файловый вход как путь нулевой настройки.
- **База знаний — гибрид на SQLite, не AnythingLLM как ядро.** Markdown-файлы — источник правды (git-friendly), индекс — SQLite: FTS5 (BM25) + sqlite-vec (векторы), слияние результатов. Ноль серверов, кроссплатформенно. Код проекта — ещё один индексируемый источник (флаг в настройках). AnythingLLM — опциональный адаптер, не зависимость.
- **Контракты между сервисами — три JSON-схемы, фиксируются сразу:** `Transcript` (сегменты: start/end/speaker/text), `Task` (title/description/assignee/SP/source/evidence), `TokenUsage` (tokens/cost_usd/billing/task_id/member_id). Каждый сервис = pip-пакет: библиотека + CLI (`uvx ...`), Docker не обязателен.
- **Meldlane (этот репозиторий) становится оркестратором:** состав команды, capacity-дашборд, склейка сервисов. Это и есть продукт.

## Сервисы

### 1. meldlane-transcribe (слой 1 — транскрибация) — итерация 1
Аудиопоток или файл → структурированный транскрипт с таймлайном и спикерами (JSON).
- Файловый вход: mp3, wav, m4a, ogg, aac (faster-whisper декодирует через PyAV, без внешнего ffmpeg) + видеоконтейнеры mp4/mkv/webm/mov/avi (опционально, через ffmpeg). Форматы — из MOM и MeetingScribe.
- Живой захват: mic + system audio, кроссплатформенный каскад (см. решения), graceful start/stop (перенести из текущего capture).
- Спикеры: канальная диаризация mic/system из коробки; pyannote — extra.
- Хранение: `outputs/YYYY-MM-DD_HH-MM/` → `transcript.json` (+ `audio.m4a` по флагу), временные файлы удаляются.
- `doctor`: какие устройства найдены, какая стратегия захвата выбрана, что делать, если системного звука нет.

### 2. llm-gateway (слой 4 — библиотека) — итерация 2
Единая точка вызова LLM: OpenRouter/OpenAI API + claude/codex CLI (подписка), строгий JSON-выход, таймауты, ретраи, единый TokenUsage (api vs subscription). Выносится из `agents/cli_runner.py`, `metrics/logger.py`, промпт-примитивов extractor.

### 3. meldlane-tasks (слой 3 — задачи) — итерация 3
Интерфейс `TaskSink` + адаптеры: markdown (backlog.md), SQLite, Plane, Jira, YouTrack — выбор в настройках. Схема Task готова. Агент — равноправный assignee.

### 4. meldlane-agents (слой 5 — агенты-исполнители) — итерация 4
Watcher: видит новые незавершённые задачи в текущем спринте/цикле → план → одобрение (ConfirmGate/Telegram или авто-режим из настроек) → исполнение через llm-gateway → testing. Ядро уже есть: `agents/executor.py` + `agents/confirm.py`.

### 5. meldlane-kb (слой 2 — база знаний) — итерация 5
Вики/требования/бэклог/техдолг/спеки → гибридный индекс (FTS5+векторы) → поиск и обогащение для остальных сервисов. Самый исследовательский — делаем, когда 1–4 уже приносят пользу.

## Порядок итераций

1 (transcribe) → 2 (llm-gateway) → 3 (tasks) → 4 (agents) → 5 (kb). Каждая итерация заканчивается рабочим релизом на GitHub, Meldlane-оркестратор переключается на сервис как на зависимость. Задачи текущей итерации — в [backlog.md](backlog.md), секция «Спринт 1».
