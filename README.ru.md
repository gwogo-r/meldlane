# Meldlane

[English version](README.md)

![Python](https://img.shields.io/badge/python-3.12%2B-blue) ![Status](https://img.shields.io/badge/status-MVP-yellow)

Трекер задач для команд, где люди и AI-агенты работают в одном списке задач — не Jira/Linear с прикрученной «AI-фичей», а трекер, где **сама ёмкость команды — это люди (story points) + агенты (токены и реальные деньги)** на одном графике.

Задачи возникают из митингов автоматически: запись → транскрибация → LLM извлекает action items → каждая задача назначается на человека **или** на агента. Задачи на агента не просто планируются — `Claude Dev` и `Codex Dev` реально запускаются как подпроцессы (Claude Code CLI / Codex CLI) с настоящим доступом к файловой системе, списываясь с подписки, а не с API-ключа по цене за токен. Перед необратимым действием — подтверждение человеком.

Построено поверх [Plane](https://plane.so) — ядро трекера не переписывается заново, только идемпотентно синхронизируется.

## Зачем

- **Агент — полноправный участник модели данных**, а не прикрученная интеграция. Один и тот же тип `Member` описывает и Романа, и Claude Dev; одна и та же `Task` назначается на любого из них.
- **Ёмкость, которая честно отражает стоимость.** У людей — story points, у агентов — токены и стоимость, разделённая на `api` (реальная оплата за токен) и `subscription` (cost-эквивалент под подпиской Claude Pro/Max или ChatGPT Plus — не отдельный счёт).
- **Реальное исполнение, а не имитация планирования.** Задачи на агента, дошедшие до CLI-провайдера, реально выполняются в изолированной копии рабочей директории и производят настоящие изменения файлов — проверено вживую, не смоделировано.
- **Честность про пробелы.** Plane ожидает, что `assignee` — реальный аккаунт пользователя воркспейса, поэтому «агент как нативный assignee» не решён: имя агента/стоимость/story points идут в описание issue текстом. Сейчас однопользовательский режим; Telegram-подтверждение написано, но вживую ещё не прогонялось.

## Как это работает

```
митинг → mic+system audio → транскрипт → LLM извлекает задачи → назначение (человек | агент)
                                                                              │
                                                агент берёт задачу ───────────┘
                                                        │
                                       план → подтверждение (human-in-the-loop) → реальное исполнение
                                                        │
                                                    testing → done
```

Каждый LLM-вызов и каждый запуск агента логирует токены/стоимость с привязкой к задаче — поэтому `capacity` показывает честную картину: кто (или что) на самом деле тратит время и деньги.

## Что в этом репозитории

- **CLI** (`main.py`) — весь конвейер: извлечь задачи из транскрипта, запустить агента на задаче, синхронизировать с Plane, поискать по базе знаний.
- **Web UI** (`api/` + `frontend/`) — read-only FastAPI-бэкенд и React-фронтенд для просмотра задач и ёмкости команды не только из терминала.
- **База знаний** (`kb/`) — полнотекстовый поиск (SQLite FTS5) по докам и коду самого проекта, чтобы агент, работающий над задачей, мог посмотреть контекст, а не гадать.

## Собрано как модульный тулчейн

Захват и транскрибация звука, LLM-вызовы и синк с Plane — каждый вынесен в отдельный опубликованный, самостоятельно полезный пакет; этот репозиторий — оркестратор, который их связывает:

- [meldlane-transcribe](https://github.com/gwogo-r/meldlane-transcribe) — mic + системный звук (WASAPI loopback на Windows, без VB-Cable) или файл на входе, структурированный транскрипт с метками спикеров на выходе.
- [llm-gateway](https://github.com/gwogo-r/llm-gateway) — один вызов и для OpenRouter/OpenAI chat completions, и для запуска Claude Code / Codex как реальных подпроцессов-агентов, с единой моделью `TokenUsage`, которая отслеживает тип оплаты.
- [meldlane-tasks](https://github.com/gwogo-r/meldlane-tasks) — идемпотентный синк `Task` → issue в Plane (запушь одну и ту же задачу дважды — получишь один issue, не два).

## Статус

Это рабочий MVP, не отполированный продукт:

- Извлечение задач из митинга, исполнение агентом, синк с Plane и учёт ёмкости — всё проверено вживую (реальный звук, реальные LLM-вызовы, реальный инстанс Plane, реальная запись файлов CLI-агентом).
- Web UI новый: оба API-эндпоинта отдают корректные данные, фронтенд собирается и dev-сервер отдаёт приложение — в реальном браузере пока не проверено.
- Telegram-подтверждение написано, но не проверено вживую; реальное исполнение агентом сейчас прогоняется только с пропуском подтверждения, в одноразовых рабочих копиях.
- Настоящей песочницы вокруг исполнения агента нет — рабочая директория CLI-агента это соглашение, не файловый джейл. Пока решается процедурно, не технически.

## Установка и запуск

```bash
pip install -r requirements.txt
cp .env.example .env   # впиши свои LLM/Plane/Telegram ключи по необходимости

python main.py members              # состав команды: люди + агенты
python main.py extract samples/standup.txt
python main.py tasks
python main.py capacity
python main.py sync-plane           # нужны PLANE_* в .env

python -m uvicorn api.app:app --reload   # web API, порт 8000
cd frontend && npm install && npm run dev  # web UI, порт 5173
```

Python 3.12+.
